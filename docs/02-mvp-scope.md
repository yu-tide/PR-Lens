# PR Lens MVP 功能清单

## 1. 文档目的

本文档用于明确 PR Lens MVP 阶段的功能范围、优先级和边界，确保项目开发目标清晰、可交付、可演示，并避免功能范围失控。

PR Lens MVP 的核心目标不是构建完整的代码评审平台，而是完成一条稳定可运行的 AI PR Review 辅助主流程。

## 2. MVP 产品目标

用户输入一个公开 GitHub Pull Request 链接后，系统自动获取 PR 信息和代码变更，并生成结构化 Review 报告，帮助开发者快速理解变更、识别风险并获得 Review 建议。

MVP 阶段聚焦三个核心价值：

- 快速理解 PR 主要改动。
- 识别需要优先关注的风险点。
- 生成可复制或可下载的 Markdown Review 报告。

## 3. MVP 核心用户

### 3.1 代码 Reviewer

主要用户包括团队 Tech Lead、资深开发者、项目 Maintainer，以及需要审核同事 PR 的开发者。

核心需求：

- 快速理解 PR 变更内容。
- 找到风险较高的文件。
- 获得具体、可执行的 Review 建议。
- 减少阅读 diff 前的初筛成本。

### 3.2 PR 作者

次要用户包括普通开发者、实习生、新人开发者和开源贡献者。

核心需求：

- 提交 Review 前自查明显问题。
- 检查是否缺少错误处理、测试或说明。
- 根据 AI 建议完善 PR 内容。

## 4. MVP 主流程

```text
输入公开 GitHub PR 链接
-> 校验并解析链接
-> 获取 PR 基本信息
-> 获取 changed files 与 patch
-> 执行规则预检查
-> 调用 AI 生成分析结果
-> 展示 PR 摘要、风险列表和 Review 建议
-> 生成 Markdown Review 报告
-> 用户复制或下载报告
```

## 5. P0 必做功能

P0 是 PR Lens MVP 必须完成的功能。只有 P0 功能完整跑通，项目才具备可演示和可交付能力。

### 5.1 GitHub PR 链接输入与校验

#### 功能说明

用户可以在首页输入公开 GitHub Pull Request 链接。

支持格式：

```text
https://github.com/{owner}/{repo}/pull/{number}
```

#### 功能要求

- 支持输入公开 GitHub PR 链接。
- 校验链接格式是否正确。
- 解析 `owner`、`repo`、`pullNumber`。
- 非法链接需要给出明确错误提示。

#### 验收标准

- 输入合法 PR 链接后，可以解析出仓库和 PR 编号。
- 输入错误链接时页面不崩溃。
- 错误提示清晰，例如：`请输入有效的 GitHub PR 链接`。

### 5.2 PR 基本信息获取

#### 功能说明

系统通过 GitHub API 获取 PR 基本信息。

#### 需要获取的信息

- 仓库名称。
- PR 标题。
- PR 作者。
- PR 描述。
- 源分支。
- 目标分支。
- changed files 数量。
- additions。
- deletions。

#### 验收标准

- 输入公开 GitHub PR 链接后，可以展示 PR 基本信息。
- GitHub API 请求失败时有明确错误提示。
- 不在前端暴露 GitHub Token。

### 5.3 Changed Files / Diff 获取

#### 功能说明

系统获取 PR 的变更文件列表和 patch 内容，用于后续规则预检查和 AI 分析。

#### 需要获取的信息

- 文件名。
- 文件状态：`added`、`modified`、`removed`、`renamed`。
- additions。
- deletions。
- patch 内容。

#### 验收标准

- 可以展示变更文件列表。
- 至少可以获取每个文件的文件名、状态和增删行数。
- patch 内容可以进入后续规则检查和 AI 分析流程。

### 5.4 规则预检查

#### 功能说明

在调用 AI 前，系统先执行基础规则检查，用于快速识别潜在风险文件。

规则预检查只输出风险提示，不作为最终 bug 结论。

#### 初版规则

系统需要检测以下情况：

| 检查项 | 示例 |
| --- | --- |
| 是否修改依赖文件 | `package.json`、`requirements.txt`、`go.mod`、`pom.xml` |
| 是否修改配置文件 | `.env.example`、`config`、`settings`、CI 配置 |
| 是否涉及鉴权或权限逻辑 | `auth`、`token`、`permission`、`session`、`jwt` |
| 是否存在疑似硬编码敏感信息 | `api_key`、`secret`、`password`、`access_token` |
| 是否存在大量删除 | 单个文件 deletions 较多 |
| 是否新增外部调用 | `fetch`、`axios`、`request`、数据库请求 |

#### 验收标准

- 系统可以输出规则命中的风险提示。
- 风险提示可以进入最终 Review 报告。
- 页面明确区分规则提示和最终 Review 结论。

### 5.5 AI 生成 PR 变更摘要

#### 功能说明

系统根据 PR 标题、描述、文件列表和 diff 内容，生成简洁的 PR 变更摘要。

#### 摘要需要回答

- 这个 PR 主要做了什么。
- 影响了哪些模块。
- Reviewer 应该优先关注哪里。

#### 输出要求

- 不超过 200 字。
- 不编造业务背景。
- 语言简洁清楚。
- 能帮助 Reviewer 快速理解 PR。

#### 验收标准

- 页面可以展示 AI 生成的变更摘要。
- 摘要内容与 PR 变更相关。
- AI 请求失败时页面不崩溃。

### 5.6 AI 生成风险列表

#### 功能说明

系统根据 diff、规则预检查结果和 PR 上下文，生成结构化风险列表。

#### 每条风险包含

- 文件名。
- 风险等级：`High`、`Medium`、`Low`。
- 风险标题。
- 风险原因。
- 建议处理方式。
- 是否需要人工确认。

#### 风险等级说明

| 等级 | 说明 |
| --- | --- |
| High | 可能影响安全、鉴权、敏感信息、核心业务逻辑或数据一致性，需要 Reviewer 优先确认。 |
| Medium | 可能影响错误处理、可维护性、测试覆盖或依赖风险，建议合并前确认。 |
| Low | 命名、注释、可读性或轻微结构建议，通常不应阻塞合并。 |

#### 验收标准

- 页面可以展示风险列表。
- 风险按等级区分。
- 每条风险都有具体文件指向。
- 不输出大量泛泛而谈的建议。

### 5.7 AI 生成 Review 建议

#### 功能说明

系统生成面向 Reviewer 或 PR 作者的 Review 建议。

#### 建议要求

Review 建议必须满足：

- 具体。
- 可执行。
- 指向文件或变更点。
- 说明为什么需要关注。
- 不把推测写成事实。

#### 示例

不合格建议：

```text
建议优化代码结构。
```

合格建议：

```text
src/services/auth.ts 中新增了 token 解析逻辑，但当前 patch 中没有看到空 token 或格式非法时的处理。建议补充输入校验，并增加对应测试用例。
```

#### 验收标准

- 页面可以展示 Review 建议。
- 建议内容与 PR 变更相关。
- 建议可以被复制到 Review 评论中。

### 5.8 Markdown Review 报告生成

#### 功能说明

系统将分析结果生成一份结构化 Markdown Review 报告。

#### 报告内容

Markdown 报告应包含：

- PR 基本信息。
- 变更摘要。
- 重点变更文件。
- 风险列表。
- Review 建议。
- 需要人工确认的问题。
- AI 分析限制说明。

#### 验收标准

- 页面可以展示 Markdown 报告预览。
- 报告结构清晰。
- 报告内容可以复制到 GitHub Review 评论中。

### 5.9 复制 / 下载报告

#### 功能说明

用户可以复制或下载 Markdown Review 报告。

#### 功能要求

- 支持一键复制 Markdown。
- 支持下载 `.md` 文件。
- 操作后需要有状态反馈。

#### 验收标准

- 点击复制后，报告内容进入剪贴板。
- 点击下载后，可以生成 Markdown 文件。
- 复制成功后展示反馈，例如：`已复制`。

### 5.10 错误提示与加载状态

#### 功能说明

系统需要在关键流程中展示明确状态，避免用户不知道当前分析进度。

#### 加载状态

至少包含：

- 正在解析 PR 链接。
- 正在获取 PR 信息。
- 正在获取变更文件。
- 正在执行规则预检查。
- 正在生成 AI Review 报告。
- 分析完成。

#### 错误状态

至少包含：

- PR 链接格式错误。
- GitHub API 请求失败。
- AI 服务请求失败。
- 当前 PR 无法访问。
- 结果解析失败。

#### 验收标准

- 用户可以明确知道系统当前在执行哪一步。
- 出错时页面不崩溃。
- 错误信息能够指导用户下一步操作。

### 5.11 Mock 演示模式

#### 功能说明

系统必须支持 Mock 模式，用于保证本地开发和演示稳定。

#### 使用场景

Mock 模式用于：

- GitHub API 不可用时演示主流程。
- AI API 不稳定时展示结果页。
- 本地开发时降低外部服务依赖。
- 无 API Key 环境下体验主要流程。

#### 功能要求

- 首页提供 `使用示例 PR 快速体验` 按钮。
- Mock 数据能完整展示摘要、风险、建议和 Markdown 报告。
- 页面明确提示当前为 Mock 示例数据。

#### 验收标准

- 不配置 API Key，也能跑通 Mock 主流程。
- Mock 模式可以用于演示。
- Mock 模式不能伪装成真实分析结果。

## 6. P1 可选增强功能

P1 是增强体验的功能。只有在 P0 全部完成且稳定后，才考虑实现。

### 6.1 风险等级筛选

支持用户按风险等级筛选风险列表。

可选项：

- 全部。
- High。
- Medium。
- Low。

### 6.2 示例 PR 快速体验

提供一个示例 PR 链接或示例按钮，让用户快速体验完整流程。

如果 Mock 模式已实现，该功能可以与 Mock 示例复用。

### 6.3 进度状态可视化

将分析过程用步骤条展示，提升用户对分析过程的感知。

基础 loading 文案已满足 MVP，步骤条属于体验增强。

### 6.4 更好的 Diff 展示

在页面展示部分 diff 内容，并支持展开或收起。

MVP 阶段只需要展示文件列表和分析结果，完整 diff 展示不是必需功能。

### 6.5 上下文裁剪优化

对超大 PR 做更精细的上下文裁剪，优先保留风险文件和核心变更。

MVP 阶段可以先实现基础裁剪。

### 6.6 Markdown 模板优化

提供不同风格的 Review 报告模板，例如：

- 简洁版。
- 详细版。
- Reviewer 评论版。

MVP 阶段只需要一版结构化报告。

## 7. 明确不做功能

以下功能不进入 MVP 范围。

### 7.1 用户登录

MVP 阶段不做用户登录。当前主流程不依赖用户体系，登录会增加权限、会话和状态管理复杂度。

### 7.2 数据库持久化

MVP 阶段不做数据库。当前版本只处理单次即时分析，不保存历史记录。

### 7.3 私有仓库授权

MVP 阶段不支持私有仓库。私有仓库需要 OAuth、GitHub App 或 Token 授权，涉及额外权限和安全处理。

### 7.4 GitHub App

MVP 阶段不做 GitHub App。GitHub App 需要安装、权限配置和 Webhook 处理，适合作为后续扩展。

### 7.5 自动发布 Review 评论

MVP 阶段不直接向 GitHub 发布评论。系统只生成可复制的 Markdown Review 报告，由用户自行确认后粘贴。

### 7.6 自动修复代码

MVP 阶段不做自动修复代码。PR Lens 的定位是 Review 辅助工具，不是代码自动修复工具。

### 7.7 自动判断是否可以合并

MVP 阶段不判断 PR 是否可以合并。AI 分析可能存在误报或漏报，最终合并决策应由开发者和团队规则决定。

### 7.8 多平台支持

MVP 阶段仅支持 GitHub。GitLab、Gitee、Bitbucket 等平台接口不同，可作为后续扩展。

### 7.9 历史分析记录

MVP 阶段不保存历史分析记录。该能力需要数据库和用户体系支撑。

### 7.10 团队管理后台

MVP 阶段不做团队管理后台。团队规则、权限和成员管理属于后续企业化能力。

## 9. 验收标准

### 9.1 功能验收

项目提交前应满足：

- 用户可以输入 PR 链接。
- 系统可以解析 PR 链接。
- 系统可以获取公开 PR 信息。
- 系统可以获取 changed files。
- 系统可以生成 PR 摘要。
- 系统可以生成风险列表。
- 系统可以生成 Review 建议。
- 系统可以生成 Markdown 报告。
- 报告可以复制或下载。
- Mock 模式可以完整跑通。

### 9.2 工程验收

项目仓库应满足：

- main 分支保持可运行。
- README 包含完整运行说明。
- `.env.example` 存在。
- 不提交真实 API Key。
- PR 描述包含功能描述、实现思路和测试方式。
- 提交记录能够体现项目持续开发过程。

## 10. 最终边界确认

PR Lens MVP 最终只做一条主线：

```text
输入公开 GitHub PR 链接
-> 获取 PR 信息和变更文件
-> 执行规则预检查
-> 使用 AI 生成摘要、风险和建议
-> 展示结构化 Review 结果
-> 生成可复制或可下载的 Markdown 报告
```

只要该链路稳定可演示，PR Lens MVP 即达到交付目标。

所有不影响该链路稳定运行的功能，默认不进入 MVP 范围。
